# 資料來源實測紀錄

比照 `tw-momentum-scanner/docs/data-sources.md` 的風格：只記「實測過、確認過行為」的事，
不記猜測。實測日期：2026-07-16。

## 1. TWSE ISIN 上市股票清單（主要資料源）

- URL：`https://isin.twse.com.tw/isin/C_public.jsp?strMode=2`
- 格式：HTML table（**不是 JSON**）
- 編碼：宣告 `charset=MS950`，須用 `requests` 的 `resp.encoding = "cp950"` 才能正確解碼；
  用預設 utf-8 或 requests 自動偵測會亂碼。**注意：`resp.encoding = "big5"`（Python 標準
  `'big5'` codec）看似能跑，但該 codec 不含「碁」等 MS950 擴充字集字元，實測會讓
  「啟碁」「宏碁」等 13 檔股票名稱的擴充字元被 requests 靜默轉成 U+FFFD 亂碼，且不會
  拋錯，肉眼掃過去以為抓對了——這是第一輪任務真實踩到、直到概念股資料核對才發現的 bug。
  務必用 `'cp950'`（MS950 在 Python 的正確 codec 名稱），不要用 `'big5'`。
- 結構：整頁依「有價證券類型」分成多個區塊，每個區塊由
  `<td colspan=7><B>區塊名稱<B></td>` 標題列開頭，區塊內每列固定 7 欄：
  `有價證券代號及名稱`、`ISIN`、`上市日`、`市場別`、`產業別`、`CFICode`、`備註`。
- 實測抓到的區塊清單（依文件出現順序）：
  股票、上市認購(售)權證、特別股、創新板、ETF、ETN、臺灣存託憑證(TDR)、受益證券-不動產投資信託
- **「股票」與「創新板」區塊的「產業別」欄位有值**，其餘區塊（權證/ETF/特別股/TDR/REITs）
  該欄位是空字串。因此「取股票＋創新板區塊」天然就達成「排除非普通股項目」的過濾效果，
  不需要額外用代號格式判斷。
- 「創新板」（臺灣創新板）實測 28 檔，代號與「股票」區塊**無重複**，人工核對樣本
  （如 2254 巨鎧精密-創／汽車工業、6854 錼創科技-KY創／半導體業）都是真實掛牌公司、
  有真實產業別分類，本質是普通股，只是上市門檻與主板不同（成長型企業專板）。
  決定：**納入資料庫，market 標記為 'TWSE'**（見下方「過濾邏輯與理由」）。
- 「代號及名稱」欄位用**全形空格**（U+3000 `　`）分隔代號與名稱，不是半形空格，
  拆分時必須用全形空格切，否則含空白字元的公司名稱會被切壞。
- 「上市日」欄位已經是**西元年** `YYYY/MM/DD` 格式（例如 `1994/09/05`），
  與 TWSE OpenAPI 常見的民國年格式不同，**不需要**做 `_roc_to_iso` 轉換，
  只需把 `/` 換成 `-`。
- 實測抓到「股票」區塊 1052 檔，代號全數為 4 碼數字，涵蓋 2330 台積電（半導體業）、
  2454 聯發科（半導體業）、2882 國泰金（金融保險業）等，人工核對合理。

## 2. TWSE ISIN 上櫃股票清單

- URL：`https://isin.twse.com.tw/isin/C_public.jsp?strMode=4`
- 格式/編碼/區塊結構與上市頁面完全相同，區塊順序不同：
  上櫃認購(售)權證、ETF、ETN、股票、特別股、受益證券-資產基礎證券
  （**「股票」區塊不是第一個**，parser 不可假設固定順序，要用區塊標題文字比對）。
- 實測抓到「股票」區塊 891 檔，代號全數為 4 碼數字，人工核對合理
  （台股上市約 1000 出頭、上櫃約 800 出頭，891 略高但仍在合理範圍內）。

## 3. TWSE OpenAPI 上市公司基本資料（補齊數字產業別代碼）

- URL：`https://openapi.twse.com.tw/v1/opendata/t187ap03_L`
- 格式：JSON list，1089 筆（比 ISIN 頁面「股票」區塊的 1052 筆多，因為這份清單還包含
  已下市但資料尚未清除、或非「股票」分類的發行人，未逐一考證，本專案只拿它做
  `公司代號 -> 產業別` 的映射表，不當作股票清單本身的來源）。
- 欄位「產業別」是**兩碼數字字串**（例如 `"24"`），不是文字名稱，需要另外對照才知道
  `24` = 半導體業。
- 交叉比對：用 ISIN 頁面「股票」區塊的 1052 檔逐一查這份 JSON 的「產業別」代碼，
  再比對 ISIN 頁面本身給的文字產業別，**代碼與文字 1:1 完全一致、零衝突**
  （同一代碼永遠對應同一產業文字），可放心把兩個來源的欄位合併進同一列。
- 已知缺口：1052 檔中有 1 檔（實測當下為 5236 凌陽創新，2026-07-16 才剛掛牌上市）
  在這份 opendata 裡查不到，推測是新股尚未被下一次批次更新收錄。這種情況
  `industry_code` 留 `NULL`，`industry_name` 仍可用 ISIN 頁面的文字（來源較即時）。
  **不臆測代碼**，符合 `models.CollectorError` 的空資料處理哲學。

## 4. TPEx OpenAPI 上櫃公司基本資料（補齊數字產業別代碼）

- URL：`https://www.tpex.org.tw/openapi/v1/mopsfin_t187ap03_O`
- 格式：JSON list，891 筆，欄位 `SecuritiesIndustryCode`（同樣兩碼數字字串）。
- **需要瀏覽器 UA**，否則會被擋（沿用 `collectors/_http.py` 內建的 `BROWSER_UA`）。
- 交叉比對確認：TPEx 與 TWSE **共用同一套產業別數字代碼表**
  （實測 26 個共同代碼，數字↔文字完全一致，零衝突，例如兩邊代碼 `"24"` 都是半導體業）。
  但代碼範圍不完全相同：TPEx 多出 `"32"` 文化創意業、`"33"` 農業科技業（TWSE 無這兩碼），
  TWSE 有 `"01"` 水泥工業、`"08"` 玻璃陶瓷、`"09"` 造紙工業、`"11"` 橡膠工業、`"12"` 汽車工業、
  `"18"` 貿易百貨業（TPEx 無這幾碼，可能是上櫃真的沒有這些產業的公司，非資料缺陷）。
- 已知缺口：891 檔中有 1 檔（實測當下為 7814 台亞半導）在這份 opendata 裡查不到，
  處理方式同上（industry_code 留 NULL，不臆測）。

## 過濾邏輯與理由（為什麼最終是 1080 + 891 檔，而不是更多）

證交所 ISIN 頁面把「所有有價證券」都放在同一份清單裡，包括 ETF、認購/認售權證、
特別股、TDR（存託憑證）、REITs（受益證券-不動產投資信託）、創新板（門檻較寬鬆的
成長型企業專板）。判斷標準是**「是否為有真實官方產業別分類的普通股」**：

- ETF／認購認售權證／特別股／TDR／REITs：ISIN 頁面本身該列「產業別」欄位是空的，
  它們是衍生／結構型金融商品或非普通股權益，跟本專案「依官方產業別做資金流向分析」
  的目的不相容，一律排除，不寫入資料庫。
- 「股票」區塊：主板普通股，全數保留。
- 「創新板」區塊：實測後確認是**普通股**、有**真實產業別分類**、代號與主板不重複，
  只是上市門檻與主板不同（類似「同一交易所的不同板」，不是獨立交易所），故予以保留，
  `market` 欄位標記為 `'TWSE'`（不額外新增第三種 market 值，避免下游查詢邏輯要多處理
  一種特例；若未來需要區分主板/創新板，可另加欄位，目前 schema 沒有這個需求）。

最終 TWSE 檔數 = 1052（股票）+ 28（創新板）= 1080；TPEx 沒有創新板等價區塊，維持 891。

# 第二輪擴充：91 檔概念股基本面/籌碼面資料（2026-07-16 實測）

## 6. TWSE 月營收 opendata（t187ap05_L）

- URL：`https://openapi.twse.com.tw/v1/opendata/t187ap05_L`
- 格式：JSON list，實測 1065 筆（上市全部公司，含非本專案「股票」清單內的發行人）。
- 只回「最新一期全量」，無歷史或單檔查詢參數 —— 已知限制，不是 bug。
- 欄位（實測 2330 台積電樣本）：`出表日期`(民國年 YYYMMDD)、`資料年月`(民國年月 YYYMM，
  即營收所屬月份)、`公司代號`、`公司名稱`、`產業別`、`營業收入-當月營收`、
  `營業收入-上月營收`、`營業收入-去年當月營收`、`營業收入-上月比較增減(%)`、
  `營業收入-去年同月增減(%)`（**即使用者所稱「年增率」**）、`累計營業收入-當月累計營收`、
  `累計營業收入-去年累計營收`、`累計營業收入-前期比較增減(%)`、`備註`（可能為 `"-"`，
  視為無備註 -> 存 NULL）。
- 數值欄位皆為字串，需轉型；營收單位為千元（新台幣仟元），本專案原樣存 INTEGER，不換算。
- 實測 2330 樣本：`資料年月=11506`（2026-06）、`出表日期=1150714`（2026-07-14 公告）、
  當月營收 442,679,969 千元、年增率 67.87%，數字與台積電公開財報數量級一致，合理。

## 7. TPEx 月營收 opendata（mopsfin_t187ap05_O）

- 端點命名沿用 `collectors/company_info.py` 已驗證過的 TWSE `_L` → TPEx `_O` 規律

  第一次就用瀏覽器 UA 打對，未命中前先嘗試過 `https://www.tpex.org.tw/openapi/v1/opendata/mopsfin_t187ap05_O`
  （多一層 `opendata/` 路徑）確認是錯的（回傳 HTML 首頁而非 JSON），正確路徑不含
  `opendata/` 這層。
- 格式/欄位與 TWSE 完全相同（同一份「公開資訊觀測站」t187ap05 報表格式），實測 891 筆。
- 需要瀏覽器 UA（沿用 `collectors/_http.py` 的 `BROWSER_UA`），否則會被擋。
- 同樣只回「最新一期全量」，無歷史區間參數。

## 8. TDCC 集保結算所股權分散表 opendata（id=1-5）

- 主要 URL：`https://opendata.tdcc.com.tw/getOD.ashx?id=1-5`
  備援 URL：`https://smart.tdcc.com.tw/opendata/getOD.ashx?id=1-5`（同一份資料的備用網域，
  實測時未觸發，主要網域一次就成功，備援只在程式碼中作為 retriable 失敗時的 fallback）。
- 格式：CSV（非 JSON），`Content-Type: text/csv; charset=UTF-8`，requests 能自動正確辨識
  編碼（`r.encoding == 'UTF-8'`），但檔頭有 UTF-8 BOM，解析前需 `lstrip('﻿')`。
- 欄位：`資料日期`(YYYYMMDD)、`證券代號`、`持股分級`、`人數`、`股數`、`占集保庫存數比例%`。
- **每週更新一次、只回「當週最新一期」全市場快照，無歷史** —— TDCC 官方就是這樣提供，
  這是已知限制，不是本專案的 bug；實測 2026-07-16 抓到的 `資料日期` 是 `20260709`
  （前一週四）。
- 實測涵蓋全市場 **4003 個不同證券代號**（含 ETF/TDR/受益證券/認購權證等，不是只有普通股），
  呼叫端必須用自己的 stock_id 白名單過濾。
- **陷阱：證券代號欄位是固定 6 碼、右側補半形空格**（例如台積電是 `"2330  "`，被動元件
  ETF 是 `"0050  "`），不是單純 4 碼數字字串，比對 stock_id 前必須先 `.strip()`，否則
  filter 會 100% 落空（本專案第一次實測時就因為漏了 `.strip()` 而查到 91 檔全部 0 筆，
  後來才發現是補空格的問題）。
- **持股分級語意**（實測 2330 樣本核對）：1~15 為 15 個標準級距（1,000 股以下 ~
  1,000,000 股以上，台股慣稱「張」= 1,000 股，故級距 15 即「持股 1,000 張以上」）；
  級距 16 對一般股票實測恆為 `0/0/0.00`（保留用途不明，未查證，不使用）；
  **級距 17 為該證券的「合計」列**（人數/股數/比例已經是 1~16 加總好的總數，可直接當
  該證券總集保人數/總集保股數使用，不需自行加總 1~15，也避免因級距 16 語意不明而
  重複計算或漏算）。
- 本專案設計的籌碼集中度代理指標（`build_fundamentals.py` 計算）：
  `pct_gt_400zhang` = 級距 12+13+14+15 的比例合計（>400,001 股，即 >400 張）；
  `pct_gt_1000zhang` = 級距 15 單獨的比例（>1,000,000 股，即 >1000 張）。
  15 級距原始明細（人數/股數/比例）另存 `levels_json` 欄位（JSON array），保留完整資訊
  供未來需要更細緻分析時使用，不因為只算兩個代理指標就丟棄原始資料。
  實測 2330：`pct_gt_400zhang=87.74%`、`pct_gt_1000zhang=85.01%`（大股東/法人持股高度
  集中，符合台積電籌碼結構的常識預期）。

## 9. tw_cache/institutional.db（跨專案唯讀共用資產，非本專案自行採集）

- 路徑：`C:\CLAUDE\tw_cache\institutional.db`，`tw-momentum-scanner` 專案的既有資產，
  `build_institutional_summary.py` **只讀取，絕不寫入或搬動此檔案**（用 SQLite URI
  `mode=ro` 開啟，連寫入權限都沒有）。
- schema：`institutional(date, stock_id, foreign_net, trust_net, dealer_net)`，PK
  `(date, stock_id)`，另有索引 `idx_stock_date(stock_id, date)`。
- 實測涵蓋期間：`2013-01-02` 至 `2026-05-28`（**執行本專案 build 腳本當下的實際值，
  不保證等於「今天」**——2026-07-16 執行時，最新一筆已是約 1.5 個月前，代表
  tw-momentum-scanner 上一次更新這個共用資料庫是在那之後就沒再跑；本專案每次執行都用
  `SELECT MAX(date)` 查詢實際值，不假設任何固定日期）。
- **陷阱：stock_id 是 yfinance 風格代號，帶市場後綴**（TWSE 股票 `.TW`、TPEx 股票
  `.TWO`，例如 `2330.TW`、`3105.TWO`），不是本專案 `stocks` 表用的純 4 碼代號。
  比對前必須依 `stocks.market` 組出對應後綴（`build_institutional_summary.py` 的
  `_MARKET_SUFFIX = {"TWSE": ".TW", "TPEx": ".TWO"}`），直接用純代號查詢會 100% 查無資料
  （本專案第一次實測時就踩到這個陷阱：91 檔全部查詢回 0 筆，後來才發現是後綴問題）。
- **已知覆蓋限制：institutional.db 只涵蓋 tw-momentum-scanner 篩選過的動能股歷史清單
  （實測 696 檔不同 stock_id），不是全市場**。本專案 91 檔概念股中實測有 **20 檔**
  從未出現在這份共用資料庫裡（`1535`／`6279`／`6285`／`6442`／`6488`／`6531`／`6613`／
  `6640`／`6691`／`6869`／`7734`／`8027`／`8046`／`8064`／`8299`／`8383`／`8473`／
  `8936`／`8996`／`9958`，見 `HANDOFF.md`），這些股票的 `institutional_flow_summary`
  就不會有列（不是留 NULL 一整列，而是整列不存在——因為連「有沒有資料」這件事本身
  都無法確認，寧可不寫入半個假訊號）。**這不是 bug，是共用資料源本身的天生覆蓋範圍
  限制**，如果未來需要這 20 檔的法人買賣超資料，需要另外對 TWSE/TPEx 官方三大法人
  API 抓歷史（不在本次任務範圍內）。
